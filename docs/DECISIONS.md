# TempuraTune 設計判断記録

重要な設計判断や、選択肢から選んだ理由を残す。

フォーマット:
- 日付
- 判断内容
- 選択肢
- 採用理由
- 代替案を採らなかった理由

---

## 2026-04-21: アプリ名
- 判断: TempuraTune
- 選択肢: TempuraTune / OilEar / FryAI など
- 採用理由: 「天ぷら(日本文化)」+「チューニング(調律)」で
  コンセプトが伝わる。海外メディアにも刺さる名前。
- 不採用理由: 他は機能直訳的で物語性が弱い

## 2026-04-21: フレームワーク選定
- 判断: Next.js 14 (App Router)
- 選択肢: Next.js / Vite + React / プレーンReact
- 採用理由: SSR可能、Vercelデプロイ簡単、Capacitor互換
- 不採用理由: ViteはSSR弱い、プレーンReactはルーティング自前

## 2026-04-21: AI判定にGemini採用
- 判断: gemini-2.5-flash
- 選択肢: Gemini / OpenAI Realtime / Claude
- 採用理由: 音声マルチモーダル対応、コスト最安、
  generateContent で structured output との相性が良い
- 不採用理由: OpenAIは料金高、Claudeは音声非対応

## 2026-04-21: マネタイズ方針
- 判断: ポートフォリオ作品として無料公開
- 採用理由: API費用は月数千円で許容範囲、本業価値向上に直結
- 不採用理由: 揚げ物頻度が低い家庭が多く、有料化で離脱

## 2026-04-21: UI設計
- 判断: Claude Designで作成、HTMLを手動移植
- 場所: design/TempuraTune.html
- 採用理由: ビジュアルを先に固めた方が実装ブレない
- 不採用理由: コードファースト開発はバイブコーダーには厳しい

## 2026-04-21: 命名規則
- 判断: リポジトリ名は `tempura-tune`、表示名は `TempuraTune`
- 選択肢: `tempuratune` / `tempura-tune` / `TempuraTune`
- 採用理由: 物理パスはkebab-caseの方が読みやすく、表示名はブランドとしてCamelCaseが自然
- 不採用理由: `tempuratune` は単語境界が見えづらく、`TempuraTune` をパスに使うとCLI操作でぶれやすい

## 2026-04-21: Next.js導入手順
- 判断: 既存ルートに直接 `create-next-app .` はせず、一時ディレクトリで生成後に必要ファイルだけ移植
- 選択肢: 既存ルートで強行 / 一時ディレクトリ生成後に移植
- 採用理由: `CLAUDE.md` と `design/` が競合して `create-next-app .` が停止したため、既存ドキュメント構成を保ったまま安全にPhase 1を進められる
- 不採用理由: 既存ルートでの生成は競合ファイルにより進行不能だった

## 2026-04-21: Phase 2のGeminiモデル
- 判断: `/api/analyze` では `gemini-2.5-flash` を使用
- 選択肢: `gemini-2.5-flash` / `gemini-3-flash-preview` / `Gemini 3.1 Flash` 系
- 採用理由: 公式ドキュメント上で音声入力と安定利用の両方に素直に乗せやすく、structured output との相性も良い
- 不採用理由: `Gemini 3.1 Flash` は確認時点で Live/TTS 系の案内が中心で、今回の通常 `generateContent` 用モデル名としては採用しづらかった

## 2026-04-22: 音声入力UX方針
- 判断: 完全リアルタイム音声ストリーミングではなく、2秒録音を連続送信する準リアルタイム方式を採用
- 選択肢: 10秒録音後に一括送信 / 2秒録音を連続送信 / WebSocket等で完全ストリーミング
- 採用理由: 油温判定は待ち時間が短い方が体験が良く、現行の `/api/analyze` を活かしながら実装難度と体感速度のバランスが最も良い
- 不採用理由: 10秒録音はプロダクト特性に対して遅く、完全ストリーミングはこの段階では実装範囲が大きすぎる

## 2026-04-22: MVP優先の進め方
- 判断: 実音声サンプリングをPhase 3の必須条件にはせず、先にAPI連携UIのMVPを作る
- 選択肢: 実音声精度検証を先に完了 / API前提でUIのMVPを先に作る
- 採用理由: まずプロダクトとして触れる状態を作る方が価値が高く、UI・体験の検証も早く回せる
- 不採用理由: 実音声検証を先に固定すると、MVP完成前に探索コストが大きくなりやすい

## 2026-04-22: 実装順の明確化
- 判断: 実音声サンプル収集より先に、実マイク入力の準リアルタイム判定フローを実装する
- 選択肢: サンプル収集を先に実施 / 実マイク入力フローを先に実装
- 採用理由: TempuraTuneの価値は「その場で油音を聞いて判断できる体験」にあり、先にその中核体験を作る方がMVPとして正しい
- 不採用理由: サンプル収集を先にすると、体験実装より検証準備が先行してしまう

## 2026-04-23: Phase 5 UI全面刷新
- 判断: 5段階温度ゾーン（TOO_LOW/LOW/MEDIUM/HIGH/TOO_HIGH）& 3種料理プリセット方式に変更
- 選択肢: 従来の曖昧ゾーン表示を維持 / 料理プリセット × 温度ゾーン × 判定ピルの3層構造に刷新
- 採用理由: 料理種別（低温野菜/中温唐揚げ/高温天ぷら）で目標温度を決め、現在地との差分を HEAT UP/ON TARGET/COOL DOWN で示す方が実用的でわかりやすい
- 不採用理由: 汎用ゾーン名だけでは「今何をすべきか」が伝わらない

## 2026-04-23: acoustic_reasoning フィールドを削除
- 判断: APIレスポンスから `acoustic_reasoning`（音響解析の詳細テキスト）を除去
- 選択肢: 詳細テキストをUIに表示 / 除去してシンプルなJSON（current_zone + judgment のみ）に絞る
- 採用理由: UIをワードレスデザインに保つ方が体験として洗練される。技術的な解析文は一般ユーザーには不要
- 不採用理由: 表示しても読まれにくく、APIコストとレスポンスサイズが無駄に増える

## 2026-04-23: 擬音語を日本語カタカナに統一
- 判断: 油音の擬音語を「Shuwa-shuwa」等のローマ字から「シュワシュワ・ピチピチ・チリチリ」等のカタカナに変更
- 採用理由: 日本のキッチンで使うアプリとして、日本語ネイティブの表現の方が直感的。カタカナは日本語話者に視覚的にも収まりがよい

## 2026-04-23: Phase 5.5 推論方式の選定（Teachable Machine）
- 判断: Gemini API（クラウド推論）から Google Teachable Machine（オンデバイス TF.js 推論）に移行する
- 選択肢: Gemini API継続 / YAMNet等の汎用モデルをファインチューン / Teachable Machine でカスタムモデル
- 採用理由: ① APIコストゼロ・オフライン動作 ② 自分のキッチン環境の音で直接学習できるため環境特化の精度が出やすい ③ ノーコードで学習〜エクスポートが完結し、実装コストが低い
- 不採用理由: 汎用モデルは自分の鍋・コンロ・換気扇の音に対応しておらず精度が出にくい。Gemini継続はAPIコスト問題が残る

## 2026-04-23: Teachable Machine クラス設計（10°C刻り + 境界ラベル）
- 判断: TEMP_140〜TEMP_190の10°C刻り6クラス ＋ TOO_LOW ＋ TOO_HIGH（収録できた場合）＋ BACKGROUND の最大9クラス構成
- 選択肢: 5段階ゾーン（現行と同じ） / 10°C刻り温度クラス / 5°C刻り（過細分）
- 採用理由: 10°C刻りなら「約170°C」と具体的な温度を表示でき、confidence補間で針を連続的に動かせる。5°C刻りは録音難易度が高く、5段階ゾーンでは温度数値が出せない
- 不採用理由: 5段階ゾーンはPhase 5完了時点でAPIで実現済み。Teachable Machine移行のメリットを活かすなら温度数値表示が必須

## 2026-04-23: 録音時の換気扇ON方針
- 判断: 全クラスの録音中、換気扇をつけたまま録音する
- 選択肢: 換気扇OFF（クリーンな油音のみ収録） / 換気扇ON（実際の料理環境を再現）
- 採用理由: 実際に天ぷらを揚げるときは換気扇をつける。学習環境と推論環境を一致させることが精度の基本原則
- 不採用理由: 換気扇OFFで録音すると、本番使用時に換気扇ノイズが未知音として精度を下げる

## 2026-04-23: confidence補間による連続温度推定
- 判断: 最高スコアのクラス1つで温度を決めるのではなく、全クラスのconfidenceスコアの重み付き平均で推定温度を算出する
- 選択肢: topクラスの固定温度を使用 / 全クラスの重み付き平均で補間
- 採用理由: 隣接クラス（TEMP_160とTEMP_170）の境界付近では両クラスにスコアが分散する。重み付き平均を使うとメーター針が165°C付近を自然に指し、なめらかな連続表示になる
- 不採用理由: topクラスのみだと針が10°C単位でカクカク飛び、体験として不自然

## 2026-04-27: 推論ツールをTeachable MachineからEdge Impulse WebAssemblyに変更
- 判断: Edge Impulse（WebAssembly形式）に切り替え
- 選択肢: Teachable Machine TF.js / Edge Impulse WebAssembly
- 採用理由: Teachable MachineのUIが使いにくく、Edge ImpulseはDSP前処理を内蔵したWASMをブラウザ・Capacitor両対応で書き出せる
- 不採用理由: Teachable MachineはTF.js直エクスポートで統合は楽だが、学習UIの制約が大きかった

## 2026-04-27: クラス設計を4クラス（noise/LOW/MID/HIGH）に簡素化
- 判断: 10°C刻り9クラスから noise/LOW/MID/HIGH の4クラスに変更
- 選択肢: 10°C刻り / 4クラスゾーン
- 採用理由: 実際の録音でクラス間の音の差が小さく10°C刻りの学習が困難。LOW=140〜150°C・MID=160〜180°C・HIGH=190°C〜の3ゾーンで天ぷら調理の判断には十分
- 不採用理由: 10°C刻りは精度要件が高く録音難易度も高い。温度数値表示は諦める

## 2026-04-27: UI刷新 ─ モード選択廃止・アンビエント全画面表示に変更
- 判断: 料理モード選択画面を廃止し、ガイド画面＋アンビエント全画面の2画面構成に変更
- 選択肢: モード選択維持 / 廃止してシンプル化
- 採用理由: 4クラスモデルでは全モードのターゲット温度（162〜182°C）がMIDに収まり、モード選択に意味がなくなった。半円メーターも3クラスではスカスカなのでアンビエント大テキストに刷新
- 不採用理由: モード維持は判定ロジックが形骸化する
