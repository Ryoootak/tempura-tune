# Phase 3: 準リアルタイム実装設計

## 目的
実マイク入力を前提に、MVPのUIと状態遷移を固める。
実音声サンプル収集や精度検証は後段に回す。

## 重点確認
- UIに必要な状態が揃っているか
- 準リアルタイム更新時の体験に不足がないか
- 実マイク入力から判定更新までの流れが無理なく作れるか

## 状態設計

### 画面状態
- `dish_select`: 料理選択前の初期画面
- `permission_request`: マイク許可を促す状態
- `listening`: 2秒録音ループ中
- `analyzing`: 最新チャンクを送信中
- `result_ready`: 最新結果を表示中
- `error`: 権限拒否、通信失敗、API失敗など

### 温度状態
- `ぬるい`
- `適温`
- `熱すぎ`
- `unknown`: 初回判定前や結果未取得時

### UIに出す情報
- 選択中の料理と目標温度
- 現在の温度ゾーン
- 推定温度帯
- 確信度
- 最新アドバイス
- 録音中 / 送信中 / エラーの状態

## 準リアルタイムの流れ
1. ユーザーが料理を選ぶ
2. マイク許可を取る
3. `MediaRecorder` で約2秒録音する
4. 録音完了チャンクを `/api/analyze` に送る
5. 返ってきた結果でメーターと文言を更新する
6. 更新後すぐ次の2秒録音を始める

補足:
- 録音と送信は完全並列ではなく、「2秒録音 → 送信 → 次の2秒録音」にする
- まずはシンプルさを優先し、Phase 4では1本ずつ順番に回す
- 体感速度が不足したら Phase 5 で前後オーバーラップを検討する

## Phase 4実装方針
- まずは 1ページ構成で作る
- `design/TempuraTune.html` の「料理選択」と「測定中」を MVP に必要な範囲で移植する
- 料理プリセットは最低限 `天ぷら / 唐揚げ / トンカツ` を用意する
- 判定結果が来るまでメーターは `unknown` 扱いにして待機表示にする
- エラー時は録音ループを止め、再試行ボタンを出す
- 停止ボタンで `dish_select` に戻せるようにする

## 判定基準
- Phase 4のUI実装に入れるだけの状態設計が固まっていること

## 設計結論
- サンプル音声収集はMVPの前提にしない
- 実際のマイク入力をその場で2秒ずつ切り、API判定する
- 完全ストリーミングは後回し
- Phase 4では「まず動く準リアルタイム体験」を最優先にする

## 動作確認方法
```bash
curl -X POST http://localhost:3000/api/analyze \
  -F "audio=@samples/sample1.m4a"
```
